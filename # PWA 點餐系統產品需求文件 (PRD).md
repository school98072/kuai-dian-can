# PWA 點餐系統產品需求文件 (PRD)

## 1. 簡介

### 1.1 專案概述

本文件旨在詳細說明一個基於漸進式網路應用程式 (Progressive Web App, PWA) 的點餐系統的產品需求。此系統將部署在 GitHub Pages 上，無需後端資料庫，所有菜單資料和使用者自定義內容均儲存在前端瀏覽器本地。系統核心功能包括菜單展示、使用者自定義菜品、點餐車管理以及將訂單內容（包含時間戳）透過電子郵件發送給廚師。

### 1.2 目標

*   提供一個輕量級、易於部署且無需後端維護的點餐解決方案。
*   允許使用者在行動裝置上像原生應用程式一樣安裝和使用 PWA。
*   支援使用者自定義菜品，提升靈活性。
*   實現訂單資訊的即時郵件通知，簡化廚師接單流程。

### 1.3 範圍

本專案範圍僅限於前端 PWA 應用程式的開發與部署。不涉及任何後端服務、資料庫管理或支付功能。

## 2. 使用者故事

*   **作為一個顧客**，我希望能夠瀏覽菜單，以便選擇我想點的菜品。
*   **作為一個顧客**，我希望能夠點擊菜品圖片和名稱，以便將其加入點餐車。
*   **作為一個顧客**，我希望能夠自定義菜名和上傳菜品圖片，並添加相關附註，以便點選不在菜單上的菜品或特殊要求。
*   **作為一個顧客**，我希望能夠在點餐車中查看我已選擇的菜品，並調整數量。
*   **作為一個顧客**，我希望能夠點擊「下單」按鈕，以便將我的訂單發送給廚師。
*   **作為一個顧客**，我希望我的訂單能夠包含下單的時間戳，以便廚師知道訂單的時效性。
*   **作為一個顧客**，我希望能夠像應用程式一樣將點餐系統安裝到我的手機主螢幕上，並離線使用。

*   **作為一個廚師**，我希望能夠收到包含顧客訂單內容和時間戳的電子郵件，以便及時處理訂單。

## 3. 功能需求

### 3.1 菜單展示

*   系統應展示預設菜單列表，每個菜品包含圖片和名稱。
*   菜單應具有響應式設計，適應不同尺寸的螢幕。

### 3.2 菜品自定義

*   使用者應能夠點擊一個「自定義菜品」按鈕或類似入口。
*   使用者應能夠輸入自定義菜名。
*   使用者應能夠上傳本地圖片作為自定義菜品的圖片。
*   使用者應能夠為自定義菜品添加文字附註（例如：少辣、多冰）。
*   自定義菜品應能被加入點餐車。

### 3.3 點餐車管理

*   使用者點擊菜品後，應將其加入點餐車。
*   點餐車應在頁面右下角以浮動按鈕或圖示顯示，並顯示已選菜品數量。
*   點擊點餐車圖示應能展開點餐車詳情，顯示所有已選菜品、數量和附註。
*   使用者應能調整點餐車中菜品的數量。
*   使用者應能從點餐車中移除菜品。

### 3.4 下單與郵件通知

*   點餐車詳情頁面應包含一個「下單」按鈕。
*   點擊「下單」按鈕後，系統應收集點餐車中的所有菜品名稱、數量、附註以及當前時間戳。
*   系統應將上述訂單資訊格式化為易於閱讀的文字內容。
*   系統應透過第三方郵件服務（例如 EmailJS 或 Formspree）將格式化後的訂單內容發送至 `school98072@gmail.com`。
*   發送成功後，應給予使用者成功提示，並清空點餐車。

### 3.5 本地儲存

*   所有菜單資料（包括預設菜單和使用者自定義菜品）應儲存在瀏覽器本地儲存（例如 IndexedDB 或 LocalStorage）中。
*   使用者關閉或重新開啟瀏覽器後，菜單資料應能持久化。

## 4. 技術要求

### 4.1 PWA 特性

*   **Service Worker**：用於實現離線快取、背景同步和推播通知（若未來擴展）。
*   **Web App Manifest**：提供應用程式名稱、圖示、啟動畫面等，使 PWA 可安裝到主螢幕。
*   **HTTPS**：部署在 GitHub Pages 上將自動支援 HTTPS，這是 PWA 的基本要求。

### 4.2 前端技術棧

*   **HTML5, CSS3, JavaScript**：基礎開發語言。
*   **框架/函式庫**：可考慮 React, Vue 或 Svelte 等輕量級框架，或純 JavaScript 實現。
*   **圖片處理**：前端圖片壓縮和預覽。
*   **本地儲存**：IndexedDB 或 LocalStorage API。
*   **郵件發送**：EmailJS, Formspree 或類似的靜態網站郵件服務。

### 4.3 部署

*   專案應部署在 GitHub Pages 上。
*   部署流程應盡可能自動化（例如使用 GitHub Actions）。

## 5. 非功能性需求

### 5.1 效能

*   頁面載入速度應快速，特別是首次載入後，應利用 Service Worker 實現接近即時的載入體驗。
*   應用程式應在低網路環境下保持可用性。

### 5.2 可用性 (Usability)

*   介面應直觀、易於操作，即使是初次使用的使用者也能快速上手。
*   提供清晰的視覺回饋，例如菜品加入點餐車的動畫或提示。

### 5.3 相容性

*   應支援主流現代瀏覽器（Chrome, Firefox, Safari, Edge）及其行動版本。

### 5.4 安全性

*   由於無後端，主要安全性考量在於前端資料的本地儲存和第三方郵件服務的安全性。應確保不洩露敏感資訊。

## 6. 未來潛在擴展 (Out of Scope for V1)

*   使用者認證與管理。
*   多個菜單類別或標籤。
*   支付功能整合。
*   推播通知（例如訂單狀態更新）。
*   廚師端訂單管理介面。
